# Philosophy: 開発プロセスの制度化

> 開発プロセスをコードとして書く。暗黙知を制度にする。AI の有無に関わらず機能するプロセスを、AI で加速する。

---

## 背景

ソフトウェア開発には、コードには書かれない知識がある。

「なぜこのライブラリを選んだのか」「なぜこの設計にしたのか」「レビューで何を見るべきか」「新しいメンバーはどう開発を始めるか」。これらは組織の知識であるにもかかわらず、個人の頭の中にしかない。人が抜ければプロセスも消え、チームが変われば品質もブレる。

AI が開発に参加するようになって、この問題はより見えやすくなった。

- AI に「設計して」と言っても、設計判断の基準が明文化されていなければ、毎回違う答えが返ってくる
- AI に「レビューして」と言っても、レビュー観点が属人的なら、AI もまた属人的になる
- AI を導入しても、プロセスが整理されていなければ、混乱が速くなるだけだ

問題は AI ではない。プロセスが明文化されていないことにある。

---

## 原則

### 1. 知識の制度化

開発プロセスにおける暗黙知を、明示的な仕組みに変換する。

「優秀なエンジニアの頭の中にある開発プロセス」を、ルール・スキル・フックとして記述する。特定の個人に依存する「うまいやり方」を、組織が共有できる形にすることで、以下が可能になる：

- 新しいメンバーがすぐに開発プロセスに参加できる
- 非エンジニアでもプロセスの一部を理解・操作できる
- AI エージェントが人間と同じプロセスに従って作業できる
- 人が抜けてもプロセスが残る

### 2. 意思決定の記録

コードは「何をしたか」を示す。しかし「なぜそうしたか」は残らない。

設計判断、ライブラリ選定、スコープの取捨選択、レビューでの指摘。各フェーズの意思決定とその理由を記録しておくことで、後からの変更が過去の文脈を踏まえて行える。同じ議論を繰り返さずに済む。判断の質をチームで振り返れる。

### 3. 対話を残す

AI との協働が日常になったことで、開発プロセスの形が変わった。

設計の検討、実装の試行錯誤、レビューの判断。以前はこれらの過程は頭の中で完結し、成果物としてはコードしか残らなかった。今は AI との対話の中に、設計の理由、却下した代替案、調査で分かった制約——かつて消えていた思考の過程が、テキストとして存在している。

ただし、対話はそのままでは消える。チャット履歴はセッションとともに消え、Slack のスレッドは流れる。

このフレームワークでは、対話の成果を Linear のコメント、ローカルの決定ログ、設計ドキュメントに構造化して残す。その際、Skill が記録のフォーマットを固める役割を担う。

```
/startproject    → 要件定義の対話を「プロジェクト概要書」として定型化
/team-implement  → 実装中の判断を「実装サマリー」として定型化
/team-review     → レビューの対話を「レビューレポート」として定型化
/deploy          → デプロイ情報を「デプロイ記録」として定型化
```

Skill を通すことで、AI との対話は再現可能な構造を持つドキュメントに変換される。フォーマットが決まっているから、後から読める。検索できる。比較できる。前回この機能を作ったとき、なぜこの設計にしたのかを追える。

対話はコードと並ぶ開発の一次成果物になった。それを流さずに残す仕組みが要る。

### 4. モジュラーに保つ

各プロセスは一つの責務だけを持つ。

```
/startproject    →  計画だけをやる
/team-implement  →  実装だけをやる
/team-review     →  レビューだけをやる
/deploy          →  デプロイだけをやる
```

組み合わせは自由で、全部使ってもいいし、一部だけでもいい。プロセスを大きなモノリスにしない。

これは今後、できることが広がっていっても変わらない。機能が増えても各プロセスの複雑さは増やさない。100のことができるようになっても、1つ1つはシンプルなままにする。複雑なことは、シンプルなプロセスの組み合わせで実現する。

### 5. 既存ツールの上に構築する

Linear、GitHub、Slack。これらは AI がいてもいなくても使えるツールだ。

このフレームワークは既存ツールを置き換えるのではなく、その上にプロセスを構築する。AI はツールへの記録を自動化し、ツール間の連携を加速するが、ツールそのものは人間が直接操作できる状態を保つ。

Manus や OpenClaw のように「全部を AI に丸投げする」アプローチもある。実験としては興味深いが、実際の企業の開発プロセスでは、いきなり全てを AI に任せるのは難しい。承認フロー、コンプライアンス、チームの合意。人間のプロセスは簡単には省略できない。

だからインクリメンタルな導入を前提にしている。

- Phase 0（基本ルール）だけ入れてもいい
- Phase 1（計画・TDD・設計記録）だけ入れてもいい
- Phase 3（AI 連携）まで入れれば、マルチエージェント協調が使える
- Phase 4（Agent Teams）まで入れれば、並列開発が可能になる

部分的に導入し、部分的に価値を得る。

### 6. 人間が判断し、AI が実行する

各フェーズの境界に人間の承認ゲートがある。

```
計画 → [承認] → 実装 → [確認] → レビュー → [承認] → デプロイ
```

AI は各フェーズ内で自律的に作業する。フェーズを跨ぐときは人間が方向を確認する。実行の速度と判断の責任を分離している。

## 構造

開発プロセスをプログラマブルにするために、以下の構造を使っている。

Infrastructure as Code がインフラの構築手順をコード化したように、開発の進め方をコード化する。

```
.claude/
├── rules/           # 制約条件（コーディング規約、セキュリティ要件）
├── skills/          # ワークフロー定義（計画、実装、レビュー、デプロイ）
├── hooks/           # ガードレール（ルール違反の検出、適切なツールへの誘導）
├── agents/          # エージェント定義（役割と責務）
└── docs/            # 知識の蓄積（設計決定、調査結果、ライブラリ制約）
```

これはバージョン管理できる。レビューできる。他のプロジェクトに移植できる。改善を重ねて進化させられる。

「うちのチームの開発プロセス」が、git リポジトリの中にフォーク可能な形で存在する。

## エージェント構成

すべての AI が同じ強みを持つわけではないので、役割を分けている。

| Agent | 役割 | 得意なこと |
|-------|------|-----------|
| **Claude Code** | 統括 | コードベース全体の把握、ユーザー対話、チーム管理 |
| **Codex CLI** | 設計顧問 | 深い推論、設計判断、デバッグ、トレードオフ分析 |
| **Gemini CLI** | リサーチャー | Google 検索、最新ドキュメント、マルチモーダル処理 |

Claude が全体を統括し、必要に応じて Codex に設計を相談し、Gemini に外部情報を調べさせる。Agent Teams により、これらは並列に、双方向に通信しながら作業する。

---

## 適応的実行

すべてのタスクに同じ重量のプロセスを適用する必要はない。

```
XS（タイポ修正）      → Claude が直接修正。チームもレビューも不要。
S （バリデーション追加）→ Claude が直接実装。簡易レビュー。
M （新規 API 追加）    → Codex に設計相談。2名レビュー。
L （認証システム構築）  → Agent Teams でフル並列。4名レビュー。
```

タスクの大きさを判定し、必要十分なリソースだけを使う。

## 現状と今後

現在、このフレームワークはローカル PC で動作している。Skill の詳細度や記録フォーマットの統一にはまだ課題がある。Slack 連携もこれからだ。

ただ、プロセスをコードとして書いている以上、そのプロセスは実行可能だ。エージェントの数が増えれば、同時に走るプロセスも増える。セキュリティレビューがコミットのたびに走る、ドキュメントがコードの変更に追従する、脆弱性が発見された時点でチケットが切られる——こういったことは、プロセスがシンプルでモジュラーであれば、エージェントを増やすだけで実現できる。

大事なのは、そこに至るまでの各ステップで、プロセスをシンプルに保ち続けることだと考えている。

---

## まとめ

1. 開発プロセスを明文化する — 暗黙知を制度にする
2. 意思決定を記録する — なぜそうしたかを追跡可能にする
3. 対話を残す — AI との対話を構造化して組織の資産にする
4. モジュラーに保つ — 各プロセスはシンプルに、組み合わせは自由に
5. 既存ツールの上に構築する — AI は加速器であり、ツールの置き換えではない
6. 人間が判断し、AI が実行する — 速度と責任を分離する
